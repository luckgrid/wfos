# Registry queries (jq cookbook)

The registry is a **pre-computed context cache**: one filtered query answers what a repo scan
otherwise would. `units.json` (and the other registry files) are generated by `moon run
ontarch:sync` and are compact (one record per line) so `jq` and RTK stay cheap.

Reusable filters live in [`queries/`](queries/). Run them with `jq -f` and pass the variable
the filter names. All examples below assume you are in `packages/ontarch/`.

> Generated files are gitignored. Run `moon run ontarch:sync` first to (re)produce them.

## Filter by attribute

```bash
# by kind  → ds, wfos
jq -r --arg kind workspace -f queries/by-kind.jq registry/units.json | jq -r .id

# by layer → ds, wfos
jq --arg layer application -f queries/by-layer.jq registry/units.json | jq -r .id

# by domain → ds, panoply, wfos
jq --arg domain luckgrid -f queries/by-domain.jq registry/units.json | jq -r .id

# by status → ds
jq --arg status planned -f queries/by-status.jq registry/units.json | jq -r .id
```

## Dependency relations (capabilities)

```bash
# units that REQUIRE a capability  → wfos
jq -r --arg cap proto -f queries/requires.jq registry/units.json

# units that PROVIDE a capability  → wfos
jq -r --arg cap metadata.registry -f queries/provides.jq registry/units.json

# units that DEPEND ON a unit (require one of its provided capabilities)
jq -r --arg id panoply -f queries/depends-on.jq registry/units.json
```

## Ad-hoc

```bash
# one-line kind histogram
jq -c '.summary.by_kind' registry/units.json          # {"native-toolchain":1,"workspace":2}

# everything an agent needs to act on one unit
jq '.units[] | select(.id=="wfos")' registry/units.json

# policy index (allow/block surfaces)
jq '.policies[] | {id, applies_to: (.applies_to // "*")}' registry/policies.json
```

## Token-cost note

One registry read replaces a workspace scan. To learn what `wfos` is, how to drive it, and the
rails it runs under, an agent reads **one descriptor / one filtered query** (`jq … units.json`,
a single small file). The equivalent discovery without the registry is a multi-file scan:
`moon.yml` + each `packages/*/moon.yml` + `packages/*/README.md` + native manifests
(`.prototools`, `Cargo.toml`, `package.json`) + the workspace `README.md` — easily a dozen reads
to infer the same `kind / layer / paths / entrypoints / capabilities / policy` the descriptor
states directly. The registry stays cheaper than its source because it is generated, never
hand-maintained, and compact by construction.
